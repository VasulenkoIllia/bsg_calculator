## Module Structure (Vertical Slices)

All modules follow the **routes → controller → service → repository → schema** pattern:

| Module | Routes | Controller | Service | Repository | Schema |
|--------|--------|-----------|---------|------------|--------|
| **auth** | `/api/v1/auth` | login, refresh, logout, me | tokens, cookies, bcrypt | verify by email, upsert | user + credentials |
| **users** | `/api/v1/users` | GET /, GET /:id, POST, PATCH /:id | CRUD ops, toPublic projection | DB queries | UserPublic |
| **companies** | `/api/v1/companies` | GET /, GET /:id | TTL-refresh, HubSpot sync, toPublic | list/find, upsert | CompanyPublic |
| **deals** | `/api/v1/deals` | GET /, GET /:id, GET /companies/:id/deals | TTL-refresh, toPublic | list/find, upsert | DealPublic |
| **hubspot** | `/api/v1/hubspot` | sync trigger, batch upsert | HubSpot API calls | N/A | N/A |
| **health** | `/health` | liveness + readiness probes | N/A | N/A | N/A |

## Key Modules Deep-Dive

### Auth (`server/modules/auth/`)
- **Routes (auth.routes.ts):**  
  `POST /login` (5/min limiter) → `POST /refresh` (20/min) → `POST /logout` → `GET /me` (Bearer)
- **Tokens (auth.tokens.ts):**  
  JWT access tokens (HS256). Refresh tokens are opaque random strings, hashed with SHA-256 in DB, sent as `HttpOnly` cookies.
- **Controller:** Parses login creds, calls service, sets refresh cookie.
- **Service:** Delegates to repository for password verify; generates tokens on success.
- **Repository:** Email lookup, password hash check, refresh token upsert.

### Companies & Deals (`server/modules/companies/`, `server/modules/deals/`)
- **TTL-driven HubSpot refresh:**  
  When `GET /companies/:id` is called and row's `lastSyncedAt` exceeds `HUBSPOT_SYNC_TTL_SECONDS` (env var), the endpoint returns cached data AND schedules a non-blocking refetch via `scheduleTtlRefresh()`. Next request sees fresh data.
- **Service layer (ttl-refresh usage):**  
  ```ts
  // Pseudo-code:
  const row = await findCompanyById(id);
  await scheduleTtlRefresh({
    lastSyncedAt: row.lastSyncedAt,
    ttlMs: HUBSPOT_SYNC_TTL_SECONDS * 1000,
    enabled: hasHubSpotToken && !isDev,
    refresh: async () => {
      const hsData = await hubspot.getCompany(row.hubspotCompanyId);
      const mapped = mapHubspotCompanyToRow(hsData);
      await upsertCompany(mapped);
    },
    logLabel: "[companies] TTL refresh",
    logContext: { companyId: row.id }
  });
  return toPublic(row);
  ```
- **Listing:** Uses `buildPage()` helper with cursor pagination.
- **Public DTO:** Drops `hubspot_raw` JSONB blob from responses.

### HubSpot (`server/modules/hubspot/`)
- **Client (hubspot.client.ts):**  
  Wraps `@hubspot/api-client`, caches auth token, handles rate limiting.
- **Mapper (hubspot.mapper.ts):**  
  Transforms HubSpot property enums → internal `companyType`, `segmentType`, `lifecycleStage` fields.
- **Service + Routes:**  
  `POST /hubspot/sync` (admin only, Phase 9) — iterates HubSpot CRM and upserts all companies/deals.
- **Backfill Script (scripts/hubspot-backfill.ts):**  
  On startup, if `HUBSPOT_AUTO_BACKFILL=true` and companies table is empty, the backfill runs in the background while `/health` responds normally.

## Sprint 2.7 Shared Helpers (Reuse in Future Sprints)

**These three helpers are critical for Phase 3+ modules (calculator-configs, documents, listings, webhooks):**

### 1. `shared/ttl-refresh.ts` — Background Sync Scheduling
**Why:** Deduplicates the "serve stale, refresh async" pattern across companies, deals, and (future) listings.  
**Key Entry Point:** `scheduleTtlRefresh(config: TtlRefreshConfig)`

- Guard: checks if `enabled && ttlMs > 0 && ageMs >= ttlMs`.
- Queues work via `setImmediate()` (non-blocking return to caller).
- Wraps refresh callback with try/catch + structured logging.
- **Contract:** Pass `lastSyncedAt`, `ttlMs`, `enabled`, `refresh` callback, log label, and structured context.

**Use in future modules:** Any listing that syncs from an external source (HubSpot, external calculator engine, etc.) can reuse this instead of duplicating `companies.service` logic.

### 2. `shared/dto-parse.ts` — Response DTO Validation
**Why:** Separates output validation (→ 500 InternalError) from input validation (→ 400 ValidationError).  
**Key Entry Point:** `parseDtoOrInternalError<Schema>(schema, data, context)`

- Parses data against Zod schema.
- On failure: logs entire projection object + detailed error paths (operator debugging).
- Throws `InternalError` (not `ValidationError`), so error-handler returns 500 (not 4xx).
- **Contract:** Always pass a freshly-constructed DTO object (not raw DB rows), so logs don't flood with sensitive blobs.

**Use in future modules:** Wrap every `toPublic()` or similar projection with this. Protects against typos in SELECT statements or Drizzle query changes that accidentally drop fields.

### 3. `shared/build-page.ts` — Cursor Pagination
**Why:** Unifies the "fetch limit+1, determine hasMore, encode nextCursor" logic.  
**Key Entry Point:** `buildPage<TRow, TPublic>(rows, limit, toPublic, toCursor) → PageResult<TPublic>`

- Fetches `limit + 1` rows to detect if more exist.
- Projects rows → DTO inside the helper (avoids mapping twice).
- Returns `{ items, nextCursor, limit }`.
- **Contract:** Pass row type, limit, projection function, and cursor extractor.

**Use in future modules:** Any listing endpoint (calculator configs, documents, saved listings) calls repository with `limit + 1`, then passes result to `buildPage()`. Cursor encoding is in `shared/pagination.ts`.

## Error Handling Envelope

All errors are caught by `middleware/error-handler.ts` and serialized as:
```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "...",
    "details": { /* optional */ }
  }
}
```

**Error Codes (shared/errors.ts):**
- `400 VALIDATION_FAILED` — ZodError on request input.
- `400 REQUEST_PARAM_INVALID` — UUID or other param parse failure.
- `401 AUTH_INVALID_CREDENTIALS` — Wrong email or password.
- `401 AUTH_TOKEN_EXPIRED` — JWT expired or malformed.
- `401 AUTH_REQUIRED` — Missing Bearer token.
- `403 AUTH_FORBIDDEN` — Insufficient role (admin gate).
- `404 NOT_FOUND` — Resource doesn't exist.
- `409 ALREADY_EXISTS` — Unique constraint violation.
- `429 RATE_LIMITED` — Too many requests.
- `500 INTERNAL_ERROR` — Server projection mismatch or unhandled throw.

## Deployment & Startup Sequence

1. **Build:** `npm run build:server` (tsc → `dist/server/`)
2. **Entrypoint:** `node --loader tsx server/index.ts` (or `tsx server/index.ts` in dev)
3. **Startup:**
   - Load env from `.env` or Docker runtime.
   - Validate schema (crash if missing/invalid).
   - Initialize DB pool.
   - Create Express app.
   - Bind to `PORT`.
   - If `HUBSPOT_AUTO_BACKFILL=true` and companies table empty, start async backfill.
4. **Shutdown:** SIGTERM/SIGINT → drain in-flight requests (10s timeout) → close HTTP server → drain DB pool → exit.

## Key Entry Points for New Developers

1. **Understand request flow:** `app.ts` → middleware stack → `modules/*/routes.ts` → controllers → services.
2. **Add a new route:** Copy `modules/companies/` structure, mount in `app.ts`.
3. **Fetch from HubSpot:** Use `hubspot.client` + `hubspot.mapper`, call `scheduleTtlRefresh()` in service.
4. **List pagination:** Use `buildPage()` + cursor helpers in `shared/pagination.ts`.
5. **Output validation:** Wrap projections with `parseDtoOrInternalError()`.
6. **Errors:** Throw `AppError` subclasses; `error-handler` catches + serializes.
